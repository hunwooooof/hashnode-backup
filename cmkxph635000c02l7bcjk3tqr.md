## 前言

相信大家多少有遇過需求方反反覆覆的情形，且需求方有各式各樣我沒想到的問題，他們有時無法想像使用者行為，有時無法理解頁面操作邏輯，且對產品完成的樣子沒有畫面、根本不知道自己要什麼……等。常常要等到 demo 了、看到實際畫面了，才知道原來這樣操做起來很奇怪（即使我們已經在確認需求時告知會很怪！），然後又退回來請我們修改。

我曾經因為需求方對於他們自己設計的 UI 選項跟資料交互邏輯無法想像，在我們完成專案並交付 demo 畫面給對方做確認時，被另外要求要將多選的選項邏輯改成單選；改好的下一次 demo，他們又突然恍然大悟──是不是 Tabs 比較好？所以又被要求改成 Tabs。（好想打人！）

有的需求方沒有改好前不會上線，但是 demo 階段反覆大改、交期拖很長；有的則是把 demo 當空氣，先跟你說可以上，上線後才問東問西，問這裡為什麼是這個顏色、那裡為什麼用那個模板？（你的設計稿裡面沒有涵蓋這些需求啊！）

不論哪種情形發生，都免不了一而再、再而三沒有效率的修改。我開始思考，到底要怎麼讓需求方理解交付時才改動的成本有多高？當時敝部門 PM 還在開缺，而需求方都是跨部門來的，我只能先從改善自己的應對方式著手、思考如何預防應對。至於需求方可以如何更進步？這個課題就留給對方了。（如果他們有覺得改三次版很沒效率的話。）

當時改完版很無助，一轉頭馬上求助 ChatGPT 遇到這種情形可以怎麼辦，把 GPT 的建議筆記下來後，有了這篇文章的誕生。

## 溝通策略

### 量化成本並舉易懂的例子

明確地讓需求方知道改動花費的時間成本：「*把多選改成單選已經花了前後端快要一天的時間，改成 Tab 又要再花半天。這些時間原本可以用來開發其他新功能。*」、「*每次改掉重做，都是在浪費之前付出的時間。*」

也可以舉比較生活化的例子，就像裝潢一間房子，說了要做三個房間，當木工切好了隔間板，才說不要隔間要改成大通鋪，等於要把隔間打掉，重新來一次。是不是聽起來超費工？

### 加入需求凍結的流程

信件來往常常會模糊掉「需求還在討論中」跟「需求已經確定」這兩個階段。可以在流程上加入明確的「需求凍結（Requirement Freeze）」信件，書面確認需求並寫明進入開發後不得大改，若要改動，請排進下一次版本，避免需求方事後說「啊我還想一改再改」，彼此建立「修改是有代價的」的共識。

#### 範例信件

```plaintext
主旨：
    【需求確認完成】[專案名稱] – [功能名稱]

內文：
    您好，以下是針對 [功能名稱] 的最終需求確認：
    
    功能說明：
    使用情境：
    預期輸入/輸出：
    UI/UX 畫面版本：

    此版本將視為 最終確認，我們會依此進行開發。
    若後續有額外修改需求，將列為 新需求 (Change Request)，並需重新評估時程與成本。
    
    請於 [回覆期限，例如 2 個工作天內] 確認無誤，若未收到回覆，將視同需求確認完成。
    
    謝謝 🙏
```

#### 信件重點

1. 標題寫名「需求確認完成」，讓需求方知道這不是能隨便改的文件。
    
2. 條列清楚功能、情境、UI/UX 版本等，避免沒有列到的文件提供模糊空間。
    
3. 加上 Change Request 的說法，讓需求方知道之後要改可以，但需要額外排程並重新估時」。
    
4. 設定期限，防止需求方拖著不回。
    

### 其他可以對需求方說的話

「我們很樂意配合改 UI 或功能，但程式不像設計稿，推翻後要重做會有很高的成本。如果能在設計稿階段就先確認需求，改動的成本幾乎為零；但等到程式完成再改，不僅是拆掉重做，也浪費了前面開發的時間。為了讓專案進度不拖延，建議在進入開發前先確認清楚需求，這樣我們能更快、更穩定地交付成果。」

---

誰不想讓使用者有良好的體驗呢？我們也很希望產品最終能以最好的樣貌呈現出去，不論是畫面或是互動方式，因為身為前端工程師，設想使用者場景、提供順暢的操作介面並減少和使用者之間的資訊落差是非常重要的。但若每樣專案都需要經歷反覆的需求變動後才能上線，不僅耗時也傷心，久而久之團隊失去活力、開發沒有效率。

雖然有時礙於自己的角色位置，不便提出那麼多意見，提了也不一定被採納，但還是寫了這篇文章提醒自己，若有可能直接或間接面對需求方時，一定要做好需求管理，對齊雙方資訊、減少資訊落差。我們理解需求一定會有變動，但究竟可以變動到多大的程度？都是需要被規範的。
